# Как организовать репозиторий для LLM-агента: аутлайн (v2, post-roast)

## Метаданные статьи

- **Целевая аудитория:** CTO, техлиды, senior-инженеры, работающие с LLM-агентами
- **Тон:** авторитетный, практический, evidence-based. Не академический, но со ссылками.
- **Целевой объём:** 2500-3200 слов
- **Ключевой нарратив:** Каждый уровень — ЛУЧШИЙ на своём масштабе. Не лестница, а карта. Выбери правильный инструмент для задачи.
- **Структурный принцип:** асимметричная подача — тиры 0 и 4 компактнее (интуитивны), тиры 1-3 подробнее (неочевидный sweet spot).
- **Подача каждого тира:** killer-факт → определение → где сияет → transition story → порог перехода → как мигрировать (1 предложение).

## 1. Введение (~250 слов)

**Заголовок:** Пять уровней организации знаний: найдите свой

**Открытие:** Разработчик удалил 2000 строк RAG-кода — точность выросла до 94%. Он не деградировал систему, он убрал лишнюю инфраструктуру. Это не аномалия: в бенчмарке LlamaIndex 2026 filesystem-агенты (grep + чтение файлов) обыграли RAG по корректности: 8.4 vs 6.4.

**Проблема:** Большинство команд ошибаются в одну из двух сторон:
- **Недоинвестируют:** 500 документов и grep — хаос, ничего не находится
- **Переинвестируют:** 10 документов и GraphRAG — инженерный фетишизм

**Тезис:** Нет "лучшего" способа организовать знания для LLM-агента. Есть 5 уровней, каждый из которых — лучший ответ для своего масштаба. Переходить на следующий нужно только когда текущий ломается на конкретной болевой точке — и ни секундой раньше.

**Контекст 2026:** Контекстные окна достигли 1M+ токенов у всех крупных провайдеров (Claude, Gemini, Llama, GPT). Это сдвигает порог, на котором инфраструктура поиска оправдана. Многие решения 2024 года сегодня — overkill.

**Источники:**
- Paul Hoke — Deleting 2000 Lines of RAG Code (Medium)
- LlamaIndex — Filesystem Tools vs Vector Search 2026
- Codingscape, Elvex, Morph — данные по контекстным окнам 2026

## 2. Tier 0: Плоские файлы (~250 слов)

**Заголовок:** Уровень 0: Всё помещается в контекст — и это прекрасно

**Killer-факт:** NotebookLM и Claude Projects — продукты на десятки миллионов пользователей, построенные на Tier 0. Это не "пока не настроили нормально". Это архитектурное решение.

**Определение:** Все файлы целиком загружаются в контекстное окно LLM. Никакого retrieval, никакой индексации. При 20 файлах по 200 строк это ~16K токенов — 1.6% окна Claude.

**Где сияет:**
- Recall и precision = 100%, когда всё в контексте — никакой retrieval это не превзойдёт
- Нулевая задержка (RAG добавляет 50-200мс, до 41% end-to-end latency)
- Нулевая стоимость настройки и обслуживания
- Персональные проекты, прототипы, конфиги, одноразовые скрипты анализа

**Анти-паттерн:** "Если ваша база знаний < 200K токенов (~500 страниц), включите её целиком в промпт и не мучайтесь" (Ahoi Kapptn).

**Порог перехода:** Lost-in-the-middle эффект: 30%+ падение точности для информации в середине контекста (Stanford/UC Berkeley, 2023). Эффективный контекст << рекламируемого. Стоимость растёт линейно. Граница: ~20 файлов / ~50K токенов.

**Пример структуры:**
```
my-project/
  README.md
  src/
    main.py
    utils.py
    config.yaml
  docs/
    architecture.md
    api-reference.md
```

**Источники:**
- Liu et al. 2023 — Lost in the Middle (arxiv)
- Elephas — Claude Projects vs NotebookLM 2026
- Ahoi Kapptn — From Long Prompt to RAG

## 3. Tier 1: Code-first (~450 слов)

**Заголовок:** Уровень 1: Grep + CLAUDE.md — так на самом деле работают все AI-кодинг-тулы

**Killer-факт:** Claude Code не поднимает vector DB. Cursor не требует embedding pipeline для старта. Все главные AI-кодинг-инструменты 2026 года используют grep как core infrastructure. Ripgrep "тихо стал несущей инфраструктурой для того, как AI пишет код".

**Определение:** Проект имеет CLAUDE.md (или AGENTS.md, .cursorrules), который объясняет агенту структуру и конвенции кодовой базы. Агент использует grep/ripgrep для поиска, читает файлы по запросу, навигирует через естественную структуру проекта.

**Где сияет:**
- **Это индустриальный стандарт:** Claude Code (agentic search без индекса), Cursor (embedding index), Windsurf (enterprise RAG pipeline) — все начинают с grep + project docs
- **Zero infrastructure:** $0/мес vs ~$185/мес для гибридной системы. Maintenance cost = 0
- **Точное совпадение объективно лучше семантического** для идентификаторов: `ERROR_4532` в vector space неотличим от `ERROR_4533`. Grep находит ровно то, что нужно
- **AGENTS.md выиграл гонку стандартизации:** Linux Foundation (AAIF), 60K+ репо, поддержка OpenAI, Anthropic, Google, AWS, Bloomberg. CLAUDE.md за 30 минут даёт агенту 80% нужного контекста

**Таблица конфиг-файлов агентов (2026):**

| Файл | Создатель | Поддержка | Scope |
|------|-----------|-----------|-------|
| CLAUDE.md | Anthropic | Claude Code | Иерархия директорий |
| AGENTS.md | OpenAI → Linux Foundation | 60K+ репо, все основные тулы | Иерархия + override |
| .cursorrules | Cursor (deprecated) | Cursor legacy | Корень проекта |
| copilot-instructions.md | GitHub | Copilot | .github/, glob patterns |
| GEMINI.md | Google | Gemini CLI | Иерархия директорий |

**Transition story:** "У вас 300 файлов кода и grep работает отлично. Потом приходит задача: найти все требования GDPR в research notes, юридических документах и meeting transcripts. Grep по слову 'GDPR' находит 5 из 20 релевантных документов — остальные говорят о 'персональных данных', 'privacy regulation', 'обработке ПДн'. Вы осознаёте, что вам нужна не лучшая поисковая система, а структурированная навигация."

**Порог перехода:** Полисемия grep (одно слово в сотнях контекстов). Non-code knowledge не помещается. Ripgrep vs indexed search при 500K+ файлов (0.013s indexed vs 15s grep). Граница: ~500 файлов, преимущественно код.

**Как мигрировать на Tier 2:** Начните с одного CLAUDE.md или AGENTS.md — это буквально 30 минут работы и уже даёт 80% эффекта.

**Пример структуры:**
```
my-repo/
  CLAUDE.md          # инструкции агенту
  AGENTS.md          # стандартизованные правила
  src/
  tests/
  docs/
    architecture.md
    adr/
```

**Источники:**
- AGENTS.md specification (agents.md)
- Morph — Codebase Indexing
- DEV.to — Cursor vs Windsurf vs Claude Code 2026
- BuildMVPFast — Ripgrep at 10 Years
- ast-grep blog — Code Search Design Space

## 4. Tier 2: Структурированная документация (~500 слов)

**Заголовок:** Уровень 2: Docs-as-code — когда документация работает на людей и агентов из одного источника

**Killer-факт:** Stripe docs, Kubernetes docs (3000+ страниц), Django docs, Terraform docs — обслуживают миллионы разработчиков. Ни один из них не использует RAG или LLM-компиляцию. И не собирается.

**Определение:** Документация организована по типу контента с формальными шаблонами, кросс-ссылками, валидируемыми на этапе сборки, и навигационной структурой (sidebar, breadcrumbs, indexes).

**Где сияет:**
- **Двойная аудитория:** Новый член команды читает те же docs, что и AI-агент. Единый источник истины. Tier 3 wiki компилируется LLM и менее удобен для людей. Tier 4 RAG невидим до запроса. Tier 2 — sweet spot, где обе аудитории получают отличный experience
- **Build-time гарантии на масштабе:** Kubernetes валидирует каждый `glossary_tooltip term_id` при сборке. Stripe Markdoc обеспечивает структурную валидацию всех code samples. Битая ссылка = failed build, а не 404 в production
- **SEO и discoverability:** Единственный тир, индексируемый Google. Tier 3/4 невидимы для поисковиков
- **Diátaxis framework:** 4 типа контента (Tutorials, How-to, Reference, Explanation). Adopted by Cloudflare, Ubuntu, Django, Gatsby. Sequin: "перестройка docs по Diátaxis обнаружила десятки улучшений продукта"
- **ADR:** Architecture Decision Records (Nygard, 2011) — AWS: "200+ ADR улучшили командное сотрудничество и скорость принятия решений"

**Transition story:** "Ваш Tier 1 репо вырос до 50 markdown-файлов документации. Grep находит слово 'authentication' в 15 из них, но вы не знаете — какой из 15 файлов отвечает на вопрос 'как настроить OAuth?' Это tutorial? Reference? Troubleshooting? Без типизации контента вы читаете все 15. С Diátaxis — идёте в `how-to/configure-oauth.md`."

**Порог перехода:** Authoring overhead — каждый документ должен соответствовать шаблону. Нет семантического обнаружения ("найди всё про fairness в AI"). Человеческий bottleneck для классификации ("это tutorial или how-to?"). 200+ документов = bottleneck ручного поддержания. Граница: ~3000 страниц с чёткой таксономией.

**Как мигрировать на Tier 2:** Примените Diátaxis к существующим docs: рассортируйте файлы по 4 типам, добавьте навигационный index.md. Один день работы для среднего репо.

**Пример структуры:**
```
docs/
  tutorials/
    getting-started.md
    deploy-first-app.md
  how-to/
    configure-auth.md
    scale-workers.md
  reference/
    api/
    cli/
  explanation/
    architecture.md
    security-model.md
  adr/
    001-use-postgres.md
    002-event-sourcing.md
```

**Источники:**
- Diataxis.fr
- Sequin blog — We Fixed Our Docs with Diataxis
- Kubernetes content guide
- Stripe blog — Markdoc
- AWS — ADR Best Practices

## 5. Tier 3: LLM-компилируемая вики (~500 слов)

**Заголовок:** Уровень 3: Метод Карпати — LLM как библиотекарь

**Killer-факт:** 383 файла → 13 статей. 81x компрессия. 130 транскриптов совещаний → 1 дайджест на 244 строки. 503x компрессия. И это не "выжимка с потерями" — это синтез: LLM находит связи между источниками, которые человек бы пропустил.

**Определение:** Трёхслойная архитектура. `raw/` — неизменяемые источники (PDF, транскрипты, заметки). `wiki/` — LLM-сгенерированные и LLM-поддерживаемые страницы (саммари, entity pages, сравнения). `index.md` — каталог всех вики-страниц с однострочными описаниями. Три операции: Ingest (прочитать → написать → обновить индекс → обновить 10-15 связанных страниц), Query (найти → ответить → сохранить хорошие ответы), Lint (найти противоречия, orphan pages, устаревшие утверждения).

**Где сияет:**
- **Zero-infrastructure:** Весь "стек" это markdown в git. `git push` — ваш deployment pipeline. Никаких vector DB, embedding services, graph databases
- **Самовосстанавливающаяся база знаний:** Операция Lint обнаруживает устаревшие утверждения и противоречия. Tier 2 docs устаревают молча — вы узнаёте об этом когда юзер пожаловался. Tier 3 wiki сигнализирует проактивно
- **Рай для соло-исследователя:** Один человек + один LLM заменяют целую документационную команду. Стоимость: только API-вызовы ($10-50 за полную компиляцию 500 источников)
- **Старт сессии: 84% сокращение токенов** (~47K → ~7.7K) — агент начинает работу с компактного индекса, а не с сотен файлов

**Три реализации (2026):**

| Реализация | Особенность |
|-----------|-------------|
| ussumant/llm-wiki-compiler | Claude Code plugin, coverage indicators (high/medium/low) |
| atomicmemory/llm-wiki-compiler | TypeScript, structured concept extraction, orphan detection |
| xoai/sage-wiki | Go, 5-pass pipeline, BM25+vector hybrid, SQLite backend |

**MindStudio:** "Если ваша база знаний < 50,000-100,000 токенов (~150-200 страниц), нет технической причины использовать RAG."

**Transition story:** "Вы ведёте исследовательский проект: 200 papers, 50 meeting notes, 30 regulatory docs. Docs-as-code не работает — у вас нет 'tutorials' и 'how-to', у вас гетерогенная мешанина. Ручное поддержание индекса занимает 2 часа в неделю и всё равно отстаёт. Вы поручаете LLM скомпилировать wiki — за час он создаёт 15 тематических страниц со ссылками на источники, которые вы бы собирали неделю."

**Порог перехода:** Индекс превышает контекст (~500+ источников). Нет fuzzy discovery ("fairness metrics" не найдёт "equity measures"). Single-maintainer bottleneck при параллельной работе. Граница: ~500 гетерогенных источников.

**Как мигрировать на Tier 3:** Создайте `raw/` и `wiki/` директории. Добавьте CLAUDE.md с конвенциями (из гиста Карпаты). Ingestion первых 20 документов — 1 час.

**Пример структуры:**
```
knowledge-base/
  CLAUDE.md          # схема и конвенции
  index.md           # каталог с однострочными саммари
  log.md             # append-only журнал операций
  raw/               # неизменяемые источники
    paper-attention-2017.pdf
    meeting-2026-03-15.txt
    regulation-gdpr.md
  wiki/              # LLM-сгенерированный markdown
    transformer-architectures.md
    gdpr-compliance.md
    team-decisions-q1.md
```

**Источники:**
- Karpathy LLM Wiki gist
- ussumant/llm-wiki-compiler (GitHub)
- MindStudio — LLM Wiki vs RAG
- VentureBeat — Karpathy LLM Knowledge Base

## 6. Tier 4: Гибридный retrieval (~350 слов)

**Заголовок:** Уровень 4: Wiki + RAG + граф знаний — когда пропущенная информация стоит денег

**Killer-факт:** European bank: EUR 20M+ экономии за 3 года. Northwestern Mutual: время обработки кейса 45 мин → 10 мин. 85% enterprise AI-приложений используют RAG в 2026. Это не hype — это production.

**Определение:** Комбинация скомпилированной вики (Tier 3) с векторным поиском для семантического обнаружения и опционально графом знаний для реляционных запросов. Стратегии объединяются через Reciprocal Rank Fusion (RRF).

**Где сияет:**
- **Discovery of unknown unknowns:** "Найди что-то связанное с fairness" — только семантический поиск может это сделать. Все предыдущие тиры требуют знать, что ищешь
- **Multi-hop reasoning:** "Статьи автора X, цитирующие метод Y, применённые в домене Z" — только graph traversal
- **Enterprise compliance:** Permission-aware retrieval, audit trails, provenance chains — это не nice-to-have, это регуляторные требования
- **Стоимость снизилась:** LazyGraphRAG (1000x дешевле индексации vs GraphRAG). LightRAG (70-90% качества за 1/100 цены). Tier 4 стал доступнее

**Когда граф знаний оправдан vs overkill:**

| Оправдан | Overkill |
|----------|----------|
| Multi-hop reasoning | < 500 источников |
| Provenance / compliance | Нет реляционных запросов |
| 500+ источников | Бюджет < $200/мес |
| Permission-aware retrieval | Соло-мейнтейнер без graph expertise |

**Стоимость:** ~$185/мес минимум (vector DB $70 + graph DB $65 + BM25 $50). Enterprise: $200K-$500K первый год.

**Пример структуры:**
```
knowledge-platform/
  wiki/              # скомпилированная вики (Tier 3)
  raw/               # источники
  embeddings/        # векторные индексы
  graph/             # граф знаний (RDF/TTL или Neo4j)
  pipelines/
    ingest.py        # chunking + embedding + graph extraction
    query.py         # RRF fusion across strategies
  config/
    retrieval.yaml   # веса стратегий, пороги
```

**Источники:**
- Microsoft GraphRAG paper + LazyGraphRAG
- LightRAG (Medium)
- HybridRAG paper (NVIDIA/BlackRock)
- Pinecone + Neo4j

## 7. Наш опыт (~200 слов)

**Заголовок:** Как мы прошли этот путь: от flat files до wiki + ontology

**Содержание:** Краткий рассказ о нашем репозитории AI-usage-lessons — учебный курс по AI с 200+ источниками. Начинали с Tier 0 (20 файлов, всё в контексте). Быстро переросли в Tier 1 (grep по catalog/exports/). Попробовали RAG — получили 10% precision на русскоязычных запросах. Попробовали онтологию — красивая схема, 0 instance data. Реализовали Tier 3 (wiki по методу Карпаты) — 7.6x сокращение tool calls, 9/9 completeness на тестовых сценариях. Добавили RAG для семантического поиска по cross-lingual запросам — но только после того, как wiki заработала.

**Ключевой вывод:** Мы попробовали перепрыгнуть с Tier 1 на Tier 4 (ontology + RAG) — и получили красивую пустую инфраструктуру. Только когда вернулись к Tier 3 как базе и добавили Tier 4 сверху — система заработала.

**Источники:**
- Наш эпик #17 (Knowledge architecture overhaul)
- Результаты тестов: 7.6x improvement

## 8. Фреймворк принятия решений (~300 слов)

**Заголовок:** Какой уровень выбрать: 3 вопроса

**Формат:** Не бинарное дерево, а матрица по двум осям + дополнительные факторы.

**Три главных вопроса:**
1. **Сколько у вас источников?** (< 20 / 20-500 файлов кода / 20-500 документов / 500+)
2. **Какой тип контента?** (код / документация для людей / гетерогенные источники / всё вместе)
3. **Кто аудитория?** (только AI / люди + AI / enterprise с compliance)

**Матрица:**

| Масштаб \ Контент | Код | Docs для людей | Гетерогенные | Смешанный |
|-------------------|-----|---------------|-------------|-----------|
| < 20 файлов | Tier 0 | Tier 0 | Tier 0 | Tier 0 |
| 20-500 | Tier 1 | Tier 2 | Tier 3 | Tier 1 + Tier 2 docs/ |
| 500+ | Tier 1 + indexed search | Tier 2 | Tier 3→4 | Tier 3 + Tier 4 для discovery |

**Гибридные ситуации:** "200 файлов кода + 50 research papers" — не надо выбирать ОДИН тир. Код живёт в Tier 1 (grep + CLAUDE.md). Papers живут в Tier 3 (wiki). Это нормально — тиры не монопольны.

**Сводная таблица метрик:**

| Метрика | Tier 0 | Tier 1 | Tier 2 | Tier 3 | Tier 4 |
|---------|--------|--------|--------|--------|--------|
| Настройка | 0 | Часы | Дни | Часы | Дни-недели |
| Обслуживание | Нет | Низкое | Среднее | Среднее | Высокое |
| Стоимость инфра | $0 | $0 | Static gen | $0 (markdown) | $185+/мес |
| Потолок | ~50K tok | ~500 файлов | ~3000 стр. | ~500 источ. | 10K+ |
| Precision | 100% | Высокая | Высокая | Высокая | Наивысшая |
| Recall (discovery) | 100%* | Низкий | Средний | Средний | Высокий |

*в пределах контекста

**Ключевое правило:** Не переходите на следующий уровень пока не испытали конкретную боль текущего.

**Источники:**
- LaRA benchmark (ICML 2025) — LC 56.3% vs RAG 49.0%
- Сводная таблица из research part 2

## 9. Заключение (~200 слов)

**Заголовок:** Большинство из вас сейчас на Tier 1. И, скорее всего, это правильно.

**Содержание:** Провокационный финал. Большинство команд работают с LLM-агентами через grep + CLAUDE.md и думают, что им "пора настроить RAG". На самом деле, если grep работает — вы на правильном месте. Если перестал — следующий шаг скорее всего Tier 2 (структурированные docs) или Tier 3 (wiki), а не Tier 4 (RAG + граф).

**Ключевые тезисы:**
- Каждый уровень имеет вечную нишу — NotebookLM (Tier 0), все AI coding tools (Tier 1), Stripe docs (Tier 2), Karpathy wiki (Tier 3)
- Главный анти-паттерн: перепрыгивание уровней. Мы сами попробовали — получили красивую пустую инфраструктуру
- Простое правило: если не испытываете конкретную боль текущего уровня — вы на правильном уровне
- CTA: посмотрите на свой текущий проект. На каком вы тире? Испытываете ли вы его конкретные pain points? Если нет — не трогайте. Если да — следующий тир ждёт.
