# Reflection: CLAUDE.md Rules, Skills, Agents, and Workflow in Practice

**Session:** First session (2026-03-31)
**Scope:** Repository bootstrap, MCP server installation, initial document work, formal program rewrite
**Purpose:** Honest assessment of how well the governance framework defined in CLAUDE.md held up against real work

---

## Orchestration Rule

CLAUDE.md states: "Claude Code acts as planner and orchestrator only. ALL implementation work MUST be delegated to subagents."

### What Was Followed

- The formal program rewrite was delegated to a subagent (doc-editor role). This is the clearest case of the rule working as intended -- content editing went through a subagent with a structured prompt.
- Diagram creation was delegated. The roadmap `.drawio` file was generated by a subagent, not by the orchestrator directly.
- This reflection writing was delegated.

### What Was Violated

- **MCP server installation.** Every `claude mcp add`, `uvx`, `npx`, and binary download was run directly by the orchestrator via Bash commands. This is unambiguously "implementation work" -- it changed the system state, installed software, and modified configuration files.
- **Initial repo scaffolding.** All 45 files in the first commit were created directly using the Write tool, not delegated. Templates, ontology schema, workflow definitions, agent/skill files -- all written by the orchestrator.
- **`.gitignore` modifications.** Every reactive `.gitignore` update was done directly.
- **`settings.json` edits.** MCP configuration changes were made directly.

### The Blurry Line

The exception clause says "repo scaffolding may be edited directly since they are planning artifacts, not implementation." This clause was stretched to cover nearly everything in the first session:

- Is `ontology/schema.ttl` scaffolding or implementation? It defines 8 OWL classes and 7 properties. We treated it as scaffolding, but it is the project's data model -- arguably the most important implementation artifact.
- Is a SKILL.md file scaffolding or implementation? Each skill file describes a multi-step automated workflow. Writing the description feels like planning, but the file IS the executable spec that Claude Code will follow. It straddles the line.
- Are MCP server installations scaffolding? They are system setup, not content creation. But they change the runtime environment, which is infrastructure work, not planning.

### Assessment

The orchestration rule is well-suited for content work: editing Google Docs, creating slides, writing lecture materials. For these tasks, delegation to subagents provides a clean separation between planning ("rewrite section 3 to cover X") and execution (the actual API calls to modify the document).

The rule is poorly suited for system setup tasks. Delegating `claude mcp add workspace-mcp` to a subagent adds overhead without adding value -- there is no creative judgment involved, just command execution. The subagent would receive the exact same instructions and run the exact same commands.

**Recommendation:** Narrow the rule. "Orchestrator-only" should apply to content creation and document editing. System setup, configuration, and repo structure changes should be explicitly permitted for the orchestrator, since they require no creative delegation.

## Roast-Before-Implement Rule

CLAUDE.md states: "For non-trivial tasks, ROAST the plan before implementation."

### What Actually Happened

The roast step was **completely skipped** for every task in the session:

1. **MCP stack plan.** The plan was presented (6 servers, their purposes, installation order). The user said "go on." No roast phase. No self-critique of server choices, format coverage gaps, or authentication requirements. Had there been a roast, the XLSX/PPTX gap in local-rag might have been caught before installation, not after.

2. **Repository structure.** 45 files were planned and created without roasting the directory layout, the naming conventions, or the number of files. A roast might have asked: "Do we really need 8 skill files and 5 agent files before any of them are functional? Could we start with 2-3 and add more when needed?"

3. **Formal program rewrite.** The gap analysis identified missing content and the rewrite proceeded immediately. No roast of the gap analysis itself. No questioning of whether the identified gaps were accurate or whether the rewrite approach (section-by-section via workspace-mcp) was optimal.

4. **Ontology schema.** 8 entity types and 7 relations were defined without roasting whether all of them were needed at this stage. A roast would ask: "Can we start with Document and Lecture only, and add the rest when we have actual data to model?"

### Why It Was Skipped

Speed. The first session had a lot of ground to cover: repo setup, 6 MCP servers, document cataloging, content analysis, and a formal program rewrite. Adding a roast phase to each of these would have roughly doubled the planning time.

The user also did not request it. CLAUDE.md says the rule is "ENFORCED," but enforcement requires either the AI to self-enforce (which it did not) or the user to demand it (which they did not). In practice, "ENFORCED" rules that depend on self-enforcement are aspirational.

### What a Roast Would Have Caught

- **MCP format gaps.** "Does local-rag handle XLSX? Let me check before we install 6 servers in sequence."
- **Over-scoped scaffolding.** "Do we need impact-check and issue-from-change skills before we have a working sync pipeline? These are week-3 features being defined on day 1."
- **Formal program accuracy.** "The gap analysis was done by reading a Google Doc through workspace-mcp. Are we confident the markdown rendering preserved all content accurately? Should we cross-check against the original before rewriting?"

### Assessment

The roast rule has genuine value for content work where errors are costly and hard to undo. A bad formal program rewrite, once pushed to Google Docs, is difficult to roll back. A roast of the gap analysis could prevent incorrect content from being written.

For setup tasks (installing servers, creating directory structures), the roast adds friction without proportional benefit. The cost of a wrong MCP server choice is low -- you just uninstall it.

**Recommendation:** Make roast mandatory for content changes (documents, lectures, slides) and optional for infrastructure changes. Add a trigger: if a task involves editing a Google Doc or creating student-facing material, roast is required. If it is repo configuration, roast is encouraged but not enforced.

## Phase Gating Rule

CLAUDE.md states: "Implement -> Verify -> Gate (user approval before next phase). Never proceed to next phase without gate."

### How It Was Applied

- **Between Issue #1 and Issue #3:** Partially gated. The user saw the PR for Issue #1 and implicitly approved moving forward. But PR #2 was never merged before Issue #3 work began. This means the "gate" was a verbal approval, not a completed merge cycle.
- **Within Issue #3:** Not gated at all. The work progressed from document cataloging to gap analysis to formal program rewrite in a continuous flow. Each of these could have been a separate phase with a gate:
  - Phase 1: Catalog documents from Google Drive (gate: user confirms catalog is complete)
  - Phase 2: Analyze gaps in formal program (gate: user confirms gap list is accurate)
  - Phase 3: Rewrite formal program (gate: user reviews rewritten document)
  - Instead, all three ran as one continuous operation.

- **Formal program rewrite specifically:** The rewrite was committed and pushed without the user reviewing the actual content changes. The gap analysis identified what was missing, the subagent rewrote the document, and the session moved on. The user has not verified whether the rewrite is accurate, complete, or appropriate.

### The Risk

The formal program rewrite is the highest-stakes change in the session. It modifies a student-facing document that defines course structure, assessment criteria, and learning outcomes. Writing this without user verification at each stage is exactly the scenario phase gating is designed to prevent.

If the gap analysis was wrong (e.g., workspace-mcp's markdown rendering dropped a section, or the "missing" content was actually in an appendix), the rewrite could have introduced errors into the formal program. Without a gate, these errors propagate forward unchecked.

### Assessment

Phase gating was treated as a suggestion rather than a requirement. The pattern of "implement everything, show results at the end" is faster but riskier. For infrastructure work, this is acceptable. For content work that affects students, it is not.

**Recommendation:** Enforce phase gating strictly for any changes to Google Docs that students or stakeholders will read. The gate must be explicit: present the changes, wait for user confirmation, then proceed. For infrastructure phases, a lighter gate (status update + "continuing unless you object") is sufficient.

## Issue-Driven Workflow

### What Worked

- Issue #1 was created for repo setup and MCP installation before work began.
- Issue #3 was created for document work.
- All commits reference issue numbers in their messages.
- Issues were added to the project board.
- Labels were created (infrastructure, lecture) and applied.

### What Did Not Work

**Issue #3 scope expanded uncontrollably.** It started as "catalog documents" and grew to encompass:
- Listing Google Drive contents
- Exporting and ingesting documents into RAG
- Analyzing the formal program against the course plan
- Identifying content gaps
- Rewriting the formal program

Each of these is a distinct task that deserves its own issue:
- Issue #3: Catalog documents from Google Drive
- Issue #4: Ingest documents into local-rag
- Issue #5: Gap analysis of formal program vs. course plan
- Issue #6: Rewrite formal program to address gaps

By packing everything into Issue #3, the issue became a catch-all that lost its usefulness as a tracking unit. You cannot close Issue #3 when "catalog docs" is done but "rewrite formal program" is not.

**No Issue #2 exists.** Issue #1 was repo setup, Issue #3 was document work. What happened to Issue #2? It is the PR number (GitHub uses shared numbering for issues and PRs). This is not a problem per se, but it can confuse anyone reading the issue list.

### Assessment

The issue-driven workflow is sound in principle and was followed for issue creation and commit referencing. The weakness is scope discipline. Issues should be small enough to close in a single work session. If an issue spawns sub-tasks, those should become new issues, not scope expansions.

**Recommendation:** Add a scope check to the workflow: if an issue's work reveals additional tasks, create new issues for them immediately rather than expanding the current issue. The original issue should be closable with a clear "done" criterion.

## Branch-Per-Issue Convention

### What Happened

- `issue-1-mcp-setup` was created correctly for Issue #1.
- Issue #3 work was committed to the `issue-1-mcp-setup` branch instead of creating `issue-3-document-work` (or similar).
- PR #2 (for Issue #1) now contains commits from both Issue #1 and Issue #3.

### Why This Is Problematic

1. **PR review is harder.** A reviewer looking at PR #2 expects to see MCP setup changes. Instead, they see MCP setup plus document analysis plus formal program rewrite plus reflection files. The PR scope does not match its title.
2. **Selective revert is impossible.** If the MCP changes need to be reverted but the document work should stay, you cannot do this cleanly because they share a branch.
3. **Issue closure is ambiguous.** When PR #2 is merged, does it close Issue #1 or Issue #3 or both? The commits reference both issues, so GitHub will link both, but only Issue #1 matches the branch name.

### Root Cause

The session did not pause to merge PR #2 before starting Issue #3. The path of least resistance was to continue on the existing branch rather than switching to main, pulling, and creating a new branch. This is a workflow discipline issue, not a tooling issue.

### Assessment

This is the most straightforward rule violation in the session and the easiest to prevent. The fix is simple: finish the branch, merge the PR, then start the next branch. No exceptions, no shortcuts.

## Anti-Patterns Table: Scorecard

| Anti-Pattern | Status | Details |
|--------------|--------|---------|
| Push to main directly | VIOLATED | Initial 45-file commit went to main. Justified as bootstrap, but avoidable. |
| Work without a GitHub Issue | FOLLOWED | Issues #1 and #3 created before work. |
| Store task state only in memory | FOLLOWED | GitHub Issues used as source of truth. Task tracking via git. |
| Skip issue creation for quick tasks | FOLLOWED | No work was done without an issue. |
| Bundle risky changes together | PARTIALLY VIOLATED | MCP config + CLAUDE.md + .gitignore in same commits. Each could have been separate. |
| Make implementation changes as orchestrator | PARTIALLY VIOLATED | MCP installation and repo scaffolding done directly. Content editing was delegated. |
| Skip roast step | VIOLATED | No roast was performed at any point during the session. |
| Proceed without phase gate | PARTIALLY VIOLATED | Loose gating between issues. No gating within Issue #3's expanding scope. |

**Score: 3 fully followed, 3 partially violated, 2 fully violated out of 8.**

This is a mediocre result for a governance framework labeled "ENFORCED." The rules that were followed are the easy ones (create issues, reference them). The rules that require discipline and slow you down (roast, phase gate, branch hygiene) were the ones violated.

This is a predictable pattern: process rules that add friction are the first to be dropped under time pressure. The question is not whether the rules are good (they are), but whether they are realistic for a single-maintainer project in its first session.

## Skills Assessment

Eight skills were defined in `.claude/skills/`:

| Skill | Used? | Value Delivered | Assessment |
|-------|-------|-----------------|------------|
| sync-library | NO | None | The most critical skill for daily operations. Not built, not used. Ironic: we defined 8 skills but did not implement the one that everything else depends on. |
| catalog-docs | Partially | Low | The concept was used (listing and analyzing Drive contents), but not as an automated skill. Done manually through workspace-mcp calls. |
| extract-links | NO | None | Requires populated catalog to extract links from. Premature definition. |
| update-lecture | NO | None | No lecture content created yet. Premature definition. |
| build-deck | NO | None | No slides created yet. Premature definition. |
| diagram-refresh | Partially | Low | One diagram was created (roadmap), but not via this skill's defined workflow. |
| issue-from-change | NO | None | Requires change detection pipeline. Premature definition. |
| impact-check | NO | None | Requires populated ontology. Premature definition. |

### Key Findings

**Zero skills were used as designed.** Every SKILL.md file describes a multi-step workflow, but none were invoked as actual Claude Code skills during the session. When similar work was done (cataloging documents, creating a diagram), it was done ad-hoc through direct tool calls, not by following the skill's documented steps.

**Six out of eight skills are premature.** They define workflows for capabilities that do not exist yet (sync pipeline, change detection, ontology population). Defining them on day one is aspirational planning, not functional architecture. The SKILL.md files are essentially wish lists.

**The one skill that matters most (sync-library) was not built.** Without sync, everything downstream is manual. The daily cycle in CLAUDE.md (sync -> catalog -> extract-links -> issue-from-change -> content updates) cannot run because step 1 does not work.

**Assessment:** The skills framework (SKILL.md files in `.claude/skills/`) has potential as executable documentation, but only if the skills are actually followed during execution. In practice, the gap between "skill defined" and "skill executed" is the entire system. A SKILL.md file that nobody reads during execution is just documentation.

**Recommendation:** Delete or defer skills that have no working prerequisites. Keep sync-library and catalog-docs. Implement sync-library first. Only define new skills when their prerequisites are in place.

## Agents Assessment

Five agents were defined in `.claude/agents/`:

| Agent | Used as Subagent? | Agent .md Referenced? | Assessment |
|-------|-------------------|----------------------|------------|
| librarian | Indirectly | NO | Document export/ingest was done, but the librarian agent definition was not read or referenced when spawning subagents. |
| course-curator | Indirectly | NO | Gap analysis resembles the curator's role, but the subagent was prompted from scratch, not from the agent .md file. |
| doc-editor | YES (closest match) | NO | The formal program rewrite was delegated to a subagent whose role matches doc-editor. But the subagent prompt was custom-written, not derived from doc-editor.md. |
| deck-editor | NO | NO | No slides work in this session. |
| issue-manager | Indirectly | NO | Issues were created via `gh` CLI, not via a subagent using the issue-manager definition. |

### Key Finding

**Agent definitions were never referenced by actual subagents.** When delegation happened, the orchestrator wrote a custom prompt describing what the subagent should do. The agent .md files in `.claude/agents/` were not consulted. This means the agent definitions are disconnected from execution.

This is a design flaw. If agent definitions exist but are not used as the source of subagent prompts, they serve no functional purpose. They are documentation of intent, not executable configuration.

**Why this happens:** The Agent tool accepts a freeform prompt. There is no mechanism to say "spawn the doc-editor agent" -- you have to write the prompt yourself. The agent .md files would need to be explicitly read and included in the subagent prompt, which adds a manual step that is easy to skip.

**Recommendation:** Either:
1. Create a convention where the orchestrator reads the relevant agent .md file and includes it in the subagent prompt (adds friction but ensures consistency), or
2. Accept that agent .md files are aspirational documentation and stop pretending they are executable, or
3. Build a wrapper skill that reads the agent .md and spawns the subagent with the correct context automatically.

## Daily/Weekly Cycle Feasibility

### Daily Cycle (as defined in CLAUDE.md)

1. **Sync:** sync-library pulls changes from Google Drive -- NOT IMPLEMENTED
2. **Catalog:** catalog-docs + extract-links update index and ontology -- NOT IMPLEMENTED (done manually once)
3. **Tasks:** changes trigger issue-from-change -- NOT IMPLEMENTED
4. **Content:** update-lecture, build-deck, diagram-refresh -- NOT IMPLEMENTED (one diagram created manually)

**Zero of four daily cycle steps are automated.** The daily cycle is entirely aspirational. Every step that was performed in the session was done manually through direct MCP tool calls.

The critical blocker is step 1 (sync). Without a reliable, repeatable way to pull Google Drive changes to local files, everything downstream is manual. The sync pipeline requires:
- Listing Drive contents (workspace-mcp can do this)
- Detecting changes since last sync (no mechanism exists)
- Exporting documents to local files (workspace-mcp can export to PDF; raw format export is limited)
- Ingesting into RAG (local-rag can do this)
- Updating ontology with new entities (open-ontologies can do this)

This is approximately 2-3 more sessions of focused work.

### Weekly Cycle

- **impact-check:** Requires populated ontology with relationships. Not ready.
- **Issue triage:** Works manually via `gh` CLI or GitHub MCP. Not automated but functional.
- **Update decisions.md:** Works. Was done in this session.

### Assessment

The daily/weekly cycles were defined as target-state documentation, not as current capabilities. This is fine for planning purposes, but CLAUDE.md labels them as if they exist today. A reader of CLAUDE.md would expect these cycles to be operational. They are not.

**Recommendation:** Add a status indicator to each cycle step: "[OPERATIONAL]" or "[PLANNED]". This sets honest expectations for anyone reading CLAUDE.md (including future Claude Code sessions).

## Recommendations for CLAUDE.md Updates

Based on this session's experience, the following changes would improve CLAUDE.md's accuracy and utility:

1. **Narrow the orchestration rule.** Explicitly permit direct execution for: MCP server installation, `.gitignore` updates, `settings.json` changes, and shell commands that do not create content. Reserve delegation for content creation and document editing.

2. **Tier the roast rule.** Mandatory for student-facing content changes and document rewrites. Encouraged but optional for infrastructure and configuration. Remove the blanket "ENFORCED" label.

3. **Tier the phase gating rule.** Strict gating (explicit user approval) for content changes. Light gating (status update, proceed unless objected) for infrastructure.

4. **Add scope discipline to issue workflow.** If an issue's work reveals additional tasks, create new issues rather than expanding scope. Each issue should be closable in a single session.

5. **Remove gws and draw-mcp from the MCP stack table.** They are not installed and not needed. If they become needed later, add them back.

6. **Add `.mcp.json` as the canonical MCP config location.** The current text does not mention where MCP servers are configured. This was the biggest time sink of the session.

7. **Mark daily/weekly cycle steps with operational status.** Distinguish between "this works today" and "this is planned."

8. **Add convention: always save exports before ingestion.** This was discovered the hard way. Google Drive -> `catalog/exports/` -> `ingest_file`. Never skip the local save.

9. **Acknowledge agent/skill implementation gap.** The agent .md and SKILL.md files are documentation, not automation. Either invest in making them executable or label them as aspirational.

## Overall Verdict

CLAUDE.md is a thorough governance document that sets high standards. For a single-maintainer project in its first session, it was approximately 40% followed. The easy rules (issue creation, commit references, branch naming) were followed. The hard rules (roast, phase gating, strict delegation, branch-per-issue) were not.

This is not a failure of the rules themselves -- they describe a mature workflow that would prevent real errors. The formal program rewrite, done without roast or phase gate, is the clearest example of risk that the rules would have mitigated.

The failure is in calibration. The rules assume a project that is past the bootstrap phase, with working infrastructure and established patterns. Applying enterprise-grade governance to a project's first hour of existence creates friction that outweighs the benefit. The rules should ramp up: lighter governance during setup, full governance once the system is operational.

The skills and agents framework is the biggest gap between aspiration and reality. Eight skills defined, zero used as designed. Five agents defined, zero referenced during delegation. The framework has potential, but it needs implementation -- the definitions alone are inert.

**The single most important next step is not more rules or more definitions. It is implementing sync-library so that the daily cycle can begin. Everything else -- ontology population, change detection, impact analysis, automated cataloging -- depends on having a working sync pipeline.**
