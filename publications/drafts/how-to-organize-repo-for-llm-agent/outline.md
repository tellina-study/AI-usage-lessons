# Как организовать репозиторий для LLM-агента: аутлайн (v3)

## Метаданные статьи

- **Целевая аудитория:** CTO, техлиды, senior-инженеры, работающие с LLM-агентами
- **Тон:** авторитетный, практический, evidence-based, со ссылками на источники прямо в тексте ("как говорит X — Y")
- **Целевой объём:** 2500-3200 слов
- **Ключевой нарратив:** Агенты решают разные задачи — и для каждой задачи свой правильный подход к организации знаний. Не лестница, а карта.
- **Структурный принцип:** асимметричная подача. Каждый тир: killer-факт → определение → где сияет (с реальными примерами) → паттерны/антипаттерны (таблица) → transition story → порог перехода → практические советы (миграция + ещё 1).
- **Источники:** упоминать inline ("как показал X — Y"), не выносить в конец секции.

## 1. Введение (~300 слов)

**Заголовок:** Пять уровней организации знаний: найдите свой

**Открытие:** LLM-агенты сегодня решают радикально разные задачи. Один агент пишет код в репозитории на 10K файлов. Другой — исследует 500 научных статей. Третий — поддерживает документацию для 200 человек. Идея, что для всех этих задач нужен один и тот же подход к организации знаний — такая же ошибка, как забивать гвозди микроскопом.

**Cursor выстрелил с grep:** Пока индустрия строила сложные RAG-пайплайны и графы знаний, Cursor взлетел до $100M ARR с простым подходом — embedding index + grep по кодовой базе. Не потому что grep лучше RAG. А потому что для кода grep — правильный инструмент. Для кода.

**Проблема двух ошибок:**
- **Недоинвестируют:** 500 документов и grep — хаос, ничего не находится
- **Переинвестируют:** разработчик удалил 2000 строк RAG-кода — точность выросла до 94% (Paul Hoke, Medium). В бенчмарке LlamaIndex 2026 filesystem-агенты обыграли RAG по корректности: 8.4 vs 6.4

**Тезис:** Нет "лучшего" способа организовать знания для LLM-агента. Есть 5 уровней, каждый из которых — лучший ответ для своего типа задачи и масштаба. Переходить на следующий нужно только когда текущий ломается на конкретной болевой точке.

**Контекст 2026:** Контекстные окна достигли 1M+ токенов у всех крупных провайдеров. Как отмечает команда Codingscape в обзоре 2026: "Gemini 3, Claude 4.6, Llama 4 Scout — все предлагают миллион токенов и больше". Это сдвигает порог, на котором инфраструктура поиска оправдана.

## 2. Tier 0: Плоские файлы (~300 слов)

**Заголовок:** Уровень 0: Всё помещается в контекст — и это прекрасно

**Killer-факт:** Google NotebookLM — продукт, позволяющий загрузить до 50 источников и задавать вопросы по ним. Claude Projects (Anthropic) — фича, где вы добавляете файлы в "проект" и агент работает с ними в полном контексте. Десятки миллионов пользователей. Никакого RAG, никаких embeddings. Просто файлы в контексте. Это не MVP — это production-архитектура.

**Определение:** Все файлы целиком загружаются в контекстное окно LLM. Никакого retrieval, никакой индексации. При 20 файлах по 200 строк это ~16K токенов — 1.6% окна Claude.

**Где сияет (реальные примеры):**
- **Персональный research:** загрузил 10 papers в Claude Project — задавай вопросы, получай синтез. За 0 минут настройки
- **Прототипирование:** весь код проекта < 5 файлов. Агент видит всё. Recall и precision = 100%
- **Конфигурация и dotfiles:** 15 конфигов infra-проекта. Полный контекст, нулевая задержка
- Как пишет команда Ahoi Kapptn: "Если ваша база знаний < 200K токенов (~500 страниц), включите её целиком в промпт"

**Паттерны / Антипаттерны:**

| Паттерн | Антипаттерн |
|---------|-------------|
| Загрузить все файлы в Claude Project | Настраивать RAG pipeline для 5 документов |
| Один README + несколько конфигов | Складывать 100 файлов в контекст "про запас" |
| Простая плоская структура | Создавать иерархию папок для 10 файлов |

**Порог перехода:** Исследование Stanford/UC Berkeley (Liu et al., 2023) показало lost-in-the-middle эффект: точность падает на 30%+ когда релевантная информация в середине контекста. Отдельное исследование (arxiv 2509.21361) зафиксировало, что "эффективный контекст всех моделей оказался на 99% меньше рекламируемого" на сложных задачах. Граница: ~20 файлов / ~50K токенов.

**Практические советы:**
- **Как начать:** просто загрузите файлы в Claude Project или NotebookLM. Ничего настраивать не нужно
- **Когда переходить:** если замечаете, что агент "забывает" информацию из середины контекста или стоимость запросов растёт линейно — пора на Tier 1

**Подробнее:** [Karpathy о порогах RAG](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

**Пример структуры:**
```
my-project/
  README.md
  src/
    main.py
    utils.py
    config.yaml
  docs/
    architecture.md
    api-reference.md
```

## 3. Tier 1: Code-first (~500 слов)

**Заголовок:** Уровень 1: Grep + CLAUDE.md — так на самом деле работают все AI-кодинг-тулы

**Killer-факт:** Cursor вышел на $100M ARR. Claude Code стал стандартом для AI-driven разработки. Windsurf привлёк сотни тысяч пользователей. Ни один из них не требует от разработчика поднимать vector DB. Все используют grep как core infrastructure. Как пишет BuildMVPFast: "Ripgrep тихо стал несущей инфраструктурой для того, как AI пишет код".

**Определение:** Проект имеет CLAUDE.md (или AGENTS.md, .cursorrules), который объясняет агенту структуру и конвенции. Агент использует grep/ripgrep для поиска, читает файлы по запросу, навигирует через естественную структуру проекта (директории, импорты, naming conventions).

**Где сияет (реальные примеры):**
- **Любой software-проект с AI-кодинг-тулом:** Claude Code (agentic grep, нет pre-built индекса), Cursor (embedding index + grep), Windsurf (enterprise RAG pipeline, но начинает с grep). Все работают "из коробки" за $0 инфраструктуры
- **Open-source проекты:** React, Linux kernel, Kubernetes codebase — десятки тысяч файлов, grep работает потому что naming conventions единообразны
- **Точное совпадение:** как отмечает ast-grep blog: "grep объективно лучше семантического поиска для идентификаторов — `ERROR_4532` в vector space неотличим от `ERROR_4533`"
- **AGENTS.md стандартизирован:** Linux Foundation (AAIF), 60K+ репо, поддержка OpenAI, Anthropic, Google, AWS. Как отмечают в HumanLayer: "CLAUDE.md за 30 минут даёт агенту 80% нужного контекста"

**Проблемы Tier 0 для агентов, которые решает Tier 1:**
- На Tier 0 агент видит ВСЁ — но не знает, что важно. CLAUDE.md даёт приоритеты
- Grep позволяет агенту читать только нужные файлы, а не загружать все 500 в контекст
- Структура директорий сама по себе — навигационная карта

**Паттерны / Антипаттерны:**

| Паттерн | Антипаттерн |
|---------|-------------|
| CLAUDE.md с архитектурой и конвенциями | Положиться на то, что агент "сам разберётся" |
| Единообразные naming conventions | Разные стили именования в разных частях проекта |
| Один AGENTS.md + отдельные .md по поддиректориям | Один гигантский CLAUDE.md на 2000 строк |
| Grep для кода, идентификаторов, конфигов | Grep для поиска концепций в естественном языке |

**Transition story:** "У вас 300 файлов кода и grep работает отлично. Потом приходит задача: найти все требования GDPR в research notes, юридических документах и meeting transcripts. Grep по слову 'GDPR' находит 5 из 20 релевантных документов — остальные говорят о 'персональных данных', 'privacy regulation', 'обработке ПДн'. Вам нужна не лучшая поисковая система, а структурированная навигация."

**Порог перехода:** Полисемия grep — одно слово в сотнях контекстов. Non-code knowledge (PDF, research, regulations) не помещается в code-first модель. Graница: ~500 файлов, преимущественно код.

**Практические советы:**
- **Как мигрировать на Tier 1:** Создайте CLAUDE.md или AGENTS.md — 30 минут работы, 80% эффекта. Опишите: архитектуру, ключевые конвенции, "как запустить", "как тестировать"
- **Организация:** Держите docs/ рядом с кодом (colocation), используйте ADR для архитектурных решений

**Подробнее:** [Спецификация AGENTS.md](https://agents.md/) | [HumanLayer — Writing a good CLAUDE.md](https://www.humanlayer.dev/blog/writing-a-good-claude-md)

**Пример структуры:**
```
my-repo/
  CLAUDE.md          # инструкции агенту: архитектура, конвенции, как запустить
  AGENTS.md          # стандартизованные правила (Linux Foundation)
  src/
  tests/
  docs/
    architecture.md
    adr/
      001-use-postgres.md
```

## 4. Tier 2: Структурированная документация (~500 слов)

**Заголовок:** Уровень 2: Docs-as-code — когда документация работает на людей и агентов из одного источника

**Killer-факт:** Stripe docs, Kubernetes docs (3000+ страниц), Django docs, Terraform docs — обслуживают миллионы разработчиков. Ни один из них не использует RAG или LLM-компиляцию. И не собирается. Как отмечают в Mintlify: "В Stripe фича не считается выпущенной, пока не написана документация. Docs учитываются при промо."

**Определение:** Документация организована по типу контента с формальными шаблонами, кросс-ссылками (валидируемыми на этапе сборки), и навигационной структурой (sidebar, breadcrumbs, indexes).

**Проблемы Tier 1 для агентов, которые решает Tier 2:**
- Grep по 50 markdown-файлам находит слово 'authentication' в 15 из них — но агент не знает, какой файл отвечает на "как настроить OAuth?" vs "почему мы выбрали OAuth" vs "что делать если OAuth сломался"
- Без типизации контента агент вынужден читать все 15 файлов. С Diátaxis — сразу идёт в `how-to/configure-oauth.md`
- CLAUDE.md описывает код, но не структуру знаний. Tier 2 даёт агенту навигационную карту документации

**Где сияет (реальные примеры):**
- **Stripe:** Создали собственный Markdoc для структурной валидации docs. Интерактивные code samples с auto-inserted API keys. Docs = часть продукта
- **Kubernetes:** 3000+ страниц, 4 content types (concept/task/tutorial/reference), build-time валидация glossary terms. Bitая ссылка = failed build
- **Sequin:** "Перестройка docs по Diátaxis фреймворку обнаружила десятки улучшений продукта" — Diátaxis (4 типа: Tutorials, How-to, Reference, Explanation) принят Cloudflare, Ubuntu, Django, Gatsby
- **AWS:** "200+ Architecture Decision Records улучшили командное сотрудничество" — ADR (формат Nygard, 2011) стандартизирует фиксацию архитектурных решений

**Двойная аудитория — ключевое преимущество:** Новый член команды читает те же docs, что и AI-агент. Это остаётся верным и на Tier 3/4 — wiki тоже markdown, она читаема. Но на Tier 2 docs создаются людьми для людей, а AI получает тот же quality бесплатно. На Tier 3 docs компилируются LLM, и хотя wiki-страницы читаемы, они оптимизированы под навигацию агента, а не под human onboarding.

**Паттерны / Антипаттерны:**

| Паттерн | Антипаттерн |
|---------|-------------|
| Diátaxis: 4 типа контента (tutorial/how-to/reference/explanation) | Плоская папка docs/ без типизации |
| Build-time валидация ссылок (как в K8s) | Ручная проверка "не сломали ли ссылки" |
| ADR для архитектурных решений | Решения в Slack/email, потерянные через месяц |
| Один навигационный index.md + sidebar | Каждый документ — остров без связей |

**Порог перехода:** Authoring overhead — каждый документ должен соответствовать шаблону. Нет семантического обнаружения ("найди всё про fairness в AI"). 200+ документов = bottleneck ручного поддержания. Как отмечают исследователи PARA-метода: "Reclassification becomes a bottleneck past ~200 items."

**Практические советы:**
- **Как мигрировать на Tier 2:** Примените Diátaxis к существующим docs: рассортируйте по 4 типам, добавьте навигационный index.md. Один день для среднего репо
- **Инструменты:** Docusaurus, MkDocs, Sphinx — все поддерживают auto-generated sidebar из файловой структуры. Номерные файлы (`01-intro.md`) задают порядок без конфига

**Подробнее:** [Diátaxis framework](https://diataxis.fr/) | [Stripe Markdoc](https://stripe.dev/blog/markdoc) | [ADR на adr.github.io](https://adr.github.io/)

**Пример структуры:**
```
docs/
  index.md              # навигационный хаб
  tutorials/
    getting-started.md
    deploy-first-app.md
  how-to/
    configure-auth.md
    scale-workers.md
  reference/
    api/
    cli/
  explanation/
    architecture.md
    security-model.md
  adr/
    001-use-postgres.md
    002-event-sourcing.md
```

## 5. Tier 3: LLM-компилируемая вики (~500 слов)

**Заголовок:** Уровень 3: Метод Карпати — LLM как библиотекарь

**Killer-факт:** 383 файла → 13 статей. 81x компрессия. 130 транскриптов совещаний → 1 дайджест на 244 строки. 503x. И это не "выжимка с потерями" — LLM находит связи между источниками, которые человек бы пропустил. Как написал Карпати: "При ~100 статьях и ~400K слов способности LLM навигировать через саммари и индексные файлы более чем достаточно."

**Определение:** Трёхслойная архитектура (Andrej Karpathy, 2026):
- `raw/` — неизменяемые источники (PDF, транскрипты, заметки). Append-only: ничего не редактируем, только добавляем
- `wiki/` — LLM-сгенерированные и LLM-поддерживаемые страницы (саммари, entity pages, сравнения)
- `index.md` — каталог всех вики-страниц с однострочными описаниями. Это и есть механизм поиска: LLM сканирует индекс, находит нужную страницу, читает её

**Три операции:** Ingest (прочитать → написать wiki-страницу → обновить индекс → обновить 10-15 связанных страниц), Query (найти ответ → сохранить хорошие ответы как новые страницы), Lint (найти противоречия, orphan pages, устаревшие утверждения).

**Где сияет (реальные примеры):**
- **Рай для соло-исследователя:** Один человек + один LLM заменяют документационную команду. Наш проект AI-usage-lessons: один мейнтейнер управляет 200+ источниками через wiki
- **Самовосстанавливающаяся база:** Lint обнаруживает устаревшие утверждения и противоречия. Tier 2 docs устаревают молча — вы узнаёте когда юзер пожалуется. Wiki сигнализирует проактивно
- **Zero-infrastructure:** Весь "стек" — markdown в git. `git push` = deployment. Никаких vector DB, embedding services, graph databases
- **84% сокращение стартовых токенов:** агент начинает сессию с компактного индекса (~7.7K), а не сотен файлов (~47K) — данные ussumant/llm-wiki-compiler

**Реализации (2026):**

| Реализация | Особенность | Ссылка |
|-----------|-------------|--------|
| [ussumant/llm-wiki-compiler](https://github.com/ussumant/llm-wiki-compiler) | Claude Code plugin, coverage indicators | GitHub |
| [atomicmemory/llm-wiki-compiler](https://github.com/atomicmemory/llm-wiki-compiler) | TypeScript, concept extraction, orphan detection | GitHub |
| [xoai/sage-wiki](https://github.com/xoai/sage-wiki) | Go, 5-pass pipeline, SQLite, watch mode | GitHub |

Как отмечает MindStudio: "Если ваша база знаний < 50,000-100,000 токенов (~150-200 страниц), нет технической причины использовать RAG."

**Паттерны / Антипаттерны:**

| Паттерн | Антипаттерн |
|---------|-------------|
| raw/ append-only, wiki/ LLM-maintained | Редактировать wiki руками (сломает при перекомпиляции) |
| Один index.md с однострочными описаниями | Вложенные индексы "на будущее" при < 100 страниц |
| Инкрементальная компиляция (только изменённые) | Полная перекомпиляция 500 источников каждый раз |
| Lint после каждого Ingest | Копить 100 источников и потом компилировать разом |

**Transition story:** "Вы ведёте исследовательский проект: 200 papers, 50 meeting notes, 30 regulatory docs. Docs-as-code не работает — у вас нет 'tutorials' и 'how-to', у вас гетерогенная мешанина. Ручное поддержание индекса — 2 часа в неделю и всё равно отстаёт. Вы поручаете LLM скомпилировать wiki — за час он создаёт 15 тематических страниц со ссылками на источники, которые вы бы собирали неделю."

**Порог перехода:** Индекс превышает контекст (~500+ источников, 50K+ токенов только индекс). Нет fuzzy discovery — "fairness metrics" не найдёт "equity measures" если wiki-страница называется иначе. Граница: ~500 гетерогенных источников.

**Практические советы:**
- **Как мигрировать на Tier 3:** Создайте `raw/` и `wiki/` директории. Добавьте CLAUDE.md с конвенциями из гиста Карпати. Ingestion первых 20 документов — 1 час
- **Начните с малого:** Не компилируйте всё сразу. Ingestion по 10-20 документов за сессию. Wiki растёт органически

**Подробнее:** [Гист Карпати — LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) | [MindStudio — LLM Wiki vs RAG](https://www.mindstudio.ai/blog/llm-wiki-vs-rag-markdown-knowledge-base-comparison) | [VentureBeat — Karpathy LLM Knowledge Base](https://venturebeat.com/data/karpathy-shares-llm-knowledge-base-architecture-that-bypasses-rag-with-an)

**Пример структуры:**
```
knowledge-base/
  CLAUDE.md          # схема и конвенции (из гиста Карпати)
  index.md           # каталог с однострочными саммари
  log.md             # append-only журнал операций
  raw/               # неизменяемые источники
    paper-attention-2017.pdf
    meeting-2026-03-15.txt
    regulation-gdpr.md
  wiki/              # LLM-сгенерированный markdown
    transformer-architectures.md
    gdpr-compliance.md
    team-decisions-q1.md
```

## 6. Tier 4: Wiki + семантический поиск + граф знаний (~450 слов)

**Заголовок:** Уровень 4: Когда индекс не помещается в контекст — добавляем семантику

**Killer-факт:** Наш проект AI-usage-lessons: wiki по методу Карпати дала 7.6x сокращение tool calls и 9/9 completeness. Но когда нужно было найти "всё про AI agents" по русскоязычным документам — wiki-индекс не помог: тема упоминалась под 5 разными именами в 15 разных местах. Добавили RAG — семантический поиск нашёл то, что grep и индекс пропустили.

**Определение:** Tier 3 (wiki) + один или оба дополнительных слоя:
- **RAG (vector search):** семантический поиск по embeddings — находит "equity measures" когда ищешь "fairness metrics"
- **Граф знаний (ontology):** структурированные связи между сущностями — "статья X цитирует метод Y, применённый в домене Z"

Wiki остаётся основой — читаемой, навигируемой, markdown в git. RAG и граф — дополнительные слои поиска поверх неё. Результаты объединяются через Reciprocal Rank Fusion (RRF).

**Где сияет (реальные примеры):**
- **Агентская база знаний (наш кейс):** 200+ источников курса по AI. Wiki — основная навигация. RAG — cross-lingual поиск (русский ↔ английский). Онтология (Oxigraph) — трассировка "требование → лекция → семинар → оценка". Стоимость: $0 инфраструктуры (Oxigraph локальный, RAG через mcp-local-rag, всё в одном репо)
- **Научная библиотека:** 500+ papers по теме исследования. Wiki компилирует тематические обзоры. RAG находит релевантные papers по семантике. Граф знаний отслеживает citation chains и methodology lineage
- **Командная база знаний:** 50 человек, 3 года working knowledge — meeting notes, design docs, post-mortems, RFCs. Wiki даёт обзоры по темам, RAG находит "тот случай когда мы уже решали похожую проблему"

**Когда добавлять RAG vs граф знаний:**

| Нужен RAG когда | Нужен граф когда |
|-----------------|------------------|
| Cross-lingual поиск (RU↔EN) | Multi-hop запросы ("papers автора X → метод Y → домен Z") |
| "Найди похожее" (fuzzy discovery) | Трассировка и provenance (requirement → test → coverage) |
| Индекс wiki > 50K токенов | Агрегация ("все papers без cited-by") |
| Гетерогенные источники | Таксономии и классификации |

**Стоимость не обязательно высокая:** В нашем кейсе — $0 инфраструктуры. Oxigraph (RDF store) — бесплатный, встроенный в MCP-сервер. mcp-local-rag — локальный, бесплатный. LazyGraphRAG (Microsoft, 2026) снизил стоимость индексации в 1000x vs классический GraphRAG. LightRAG даёт 70-90% качества за 1/100 цены. Tier 4 стал доступным.

**Паттерны / Антипаттерны:**

| Паттерн | Антипаттерн |
|---------|-------------|
| Wiki как основа + RAG/граф как дополнительные слои | RAG вместо wiki (теряете навигацию и читаемость) |
| Локальные бесплатные инструменты (Oxigraph, local-rag) | Enterprise vector DB за $200/мес для 100 документов |
| Добавлять слои по одному (сначала RAG, потом граф) | Строить всю инфраструктуру сразу "на вырост" |
| Граф для конкретных relational запросов | Граф "потому что красиво" без чётких use cases |

**Практические советы:**
- **Как мигрировать на Tier 4:** Начните с RAG поверх существующей wiki. Один вечер: установить local-rag, инgestировать wiki/ папку, протестировать поиск. Граф добавляйте только когда появятся конкретные relational запросы
- **Научная библиотека:** Организуйте `raw/papers/` по темам, wiki компилирует literature reviews автоматически. RAG находит papers по семантике, а не по keywords

**Подробнее:** [Microsoft LazyGraphRAG](https://www.microsoft.com/en-us/research/blog/lazygraphrag-setting-a-new-standard-for-quality-and-cost/) | [LightRAG](https://medium.com/accelerated-analyst/lightrag-a-better-approach-to-graph-enhanced-retrieval-augmented-generation-0ac9e7bf9b74) | [Pinecone + Neo4j](https://www.pinecone.io/learn/vectors-and-graphs-better-together/)

**Пример структуры:**
```
knowledge-base/
  CLAUDE.md
  index.md           # wiki-индекс (Tier 3)
  raw/               # источники
    papers/
      by-topic/
    meeting-notes/
    regulations/
  wiki/              # LLM-компилированные страницы
  index/             # RAG-индекс (embeddings)
  ontology/          # граф знаний (TTL/RDF)
    schema.ttl       # классы и свойства
    store.ttl        # instance data
    queries/         # SPARQL-запросы
```

## 8. Фреймворк принятия решений (~250 слов)

**Заголовок:** Два вопроса, которые определяют ваш уровень

**Два ключевых вопроса:**
1. **Сколько у вас источников?** (< 20 / 20-500 / 500+)
2. **Что это — код или документация?** (код и code-adjacent docs / документация для людей / research, papers, гетерогенные источники)

**Матрица:**

| Масштаб \ Контент | Код | Документация для людей | Research, papers, mixed sources |
|-------------------|-----|----------------------|-------------------------------|
| < 20 файлов | Tier 0 | Tier 0 | Tier 0 |
| 20-500 | Tier 1 (grep + CLAUDE.md) | Tier 2 (docs-as-code) | Tier 3 (LLM wiki) |
| 500+ | Tier 1 + indexed search | Tier 2 (он масштабируется до 3000+) | Tier 3 + Tier 4 (добавить RAG/граф) |

**Гибридные ситуации нормальны:** "200 файлов кода + 50 research papers" — код на Tier 1 (grep + CLAUDE.md), papers на Tier 3 (wiki). Тиры не монопольны, они про тип контента.

**Ключевое правило:** Не переходите на следующий уровень пока не испытали конкретную боль текущего.

## 9. Заключение (~200 слов)

**Заголовок:** Большинство из вас на Tier 1. И, скорее всего, это правильно.

**Провокационный финал:** Анекдот/кейс — кто-то потратил неделю на RAG pipeline, потом узнал что Claude Code с grep решает задачу за секунды. Или наоборот — кто-то месяц грепал 500 документов, потом за час поднял wiki и сэкономил остаток квартала.

**Вывод:** Правильный ответ зависит от задачи. Tier 0 навсегда останется лучшим для малых проектов. Tier 1 навсегда — для кода. Stripe никогда не перейдёт на RAG. Karpathy wiki — идеальна для исследователей. Каждый уровень — не ступенька лестницы, а правильный инструмент для своего масштаба.

Простое правило: если не испытываете конкретную боль текущего уровня — вы на правильном месте.
